Sections below have been edited inline to reflect the locked decisions; the original-draft alternatives are preserved where they help explain *why* a path was rejected.

**Source intents:** `ArchitectureSpecification.md` §1.1 (ROMA: Atomizer/Planner/Executor/Aggregator), `LocalAgenticRAGArch.md` §"Hierarchical RL" (Wide-to-Deep planner with parallel sub-agents), `docs/03_ROADMAP.md` v0.2 deliverable 2.

## Goal

Replace the single-pass `retrieve → synth → verify → assemble` flow with a recursive **plan-then-execute** loop:

> The Atomizer decides whether the query is atomic or needs decomposition. If composite, the Planner emits a DAG of focused sub-queries. The Planner fans out to N parallel Executors (each one a self-contained retrieve+synth pass against the vault). The Aggregator merges the sub-results into a single coherent Report. The existing verifier + repair loop + assembler run once over the merged Report.

The headline value: **tighter sub-queries → tighter per-Executor retrieval → fewer irrelevant chunks in any single synth call → fewer fabricated citations.** This is the architectural cure for the "irrelevant citations" symptom observed in the v0.2.1 daily-use test.

## Non-goals (explicit deferrals)

1. **No recursion.** The spec describes ROMA as recursive — sub-tasks fed back into a child Atomizer. v0.2.2 is single-level: one Atomizer → one Planner → N flat Executors → one Aggregator. Recursion lands in a later iteration once we know the flat case behaves.
2. **No verifier-strictness or judge-prompt changes.** Per-Executor verify+repair is *in* scope (see sign-off below). What is *out* of scope is changing the judge model's prompt, the four `FailureKind` definitions, or the alignment threshold. v0.2.3 picks up that work informed by diagnostic data this plan adds.
3. **No web sources.** Sub-queries route only to the vault. Web search/fetch is deliverable 4-5, and the Planner gains a "target = vault | web" field then.
4. **No Mem0 injection into the Planner.** Memory is deliverable 3. The Planner currently has no cross-session context; in deliverable 3 we add Mem0 as a "facts the Planner already knows" pre-context.

**Sign-off note:** the original draft listed "no LLM-backed Atomizer in v0.2.2" as a non-goal. **Reversed by sign-off** — Atomizer is now an LLM node running on the judge model. See "Atomizer (LLM)" section below.

## Mapping ROMA roles to LangGraph nodes

| Spec role | v0.2.2 node | LangGraph mechanism | Cost |
|---|---|---|---|
| **Atomizer** | `atomize` | async node; one LLM call on the **judge model** returning `AtomizerVerdict{decompose: bool, rationale: str}`. Runs on every query. | 1 LLM call (small/fast model) |
| **Planner** | `plan` | async node; one LLM call on the synthesis model returning `Plan{sub_queries: list[SubQuery]}`. Skipped entirely if `decompose=False`. | 0 or 1 LLM call |
| **Executor** | `execute` | async node, fan-out target via `Send`. Each invocation = `retrieve(sub_query) → synth → verify → repair_loop → emit SubReport`. Per-Executor repair budget = `max_repair_attempts`. | 1 synth + up to N repairs + verify (per Executor) |
| **Aggregator** | `aggregate` | async node; deterministic merge of `list[SubReport]` into a single `Report`. May be promoted to an LLM call in v0.2.3 if deterministic merge produces incoherent prose. | 0 LLM calls in v0.2.2 |
| (existing) verify / repair / assemble | unchanged from v0.2.1 | runs **again** after Aggregator on the merged Report (catches cross-Report inconsistencies) | unchanged |

**Cost model — atomic path:** `atomize` (1 judge call) → `plan` skipped → one Executor (1 synth + verify + ≤2 repairs) → trivial `aggregate` passes through → post-merge verify+repair (already covered by Executor; the post-merge cycle short-circuits when sub-Report's failures list is empty). **Worst case ~v0.2.1 + 1 judge call.**

**Cost model — decomposed path (N sub-queries):** `atomize` (1) + `plan` (1) + N × [1 synth + ≤2 repair + verify] + final verify + ≤2 final repairs. With N=5 and repairs filling their budget: **up to ~20 LLM calls.** Local-hardware latency is the headline risk — see Risk R2.

## State schema diff

v0.2.1 `ResearchState` (TypedDict) gains:

```python
class AtomizerVerdict(BaseModel):
    decompose: bool
    rationale: str               # 1 sentence; logged + surfaced in execution log

class SubQuery(BaseModel):
    text: str                    # the focused sub-question
    rationale: str               # 1 sentence — why this sub-query
    target: Literal["vault"]     # extension point for v0.2.4 web routing

class Plan(BaseModel):
    sub_queries: list[SubQuery]  # min_length=1, max_length=PLANNER_FANOUT_CAP

class SubReport(BaseModel):
    sub_query: SubQuery
    chunks: list[ScoredChunk]    # what this Executor retrieved
    report: Report               # what it synthesised (one Report per Executor)
    verification: VerificationReport  # per-Executor verify outcome (sign-off)
    attempts: int                # synth + repair count for this Executor

class ResearchState(TypedDict, total=False):
    # — inputs (unchanged) —
    query, k, max_repair_attempts, skip_alignment

    # — new (Atomizer/Planner) —
    decompose_override: Literal["auto"] | bool  # from /research API
    atomizer_verdict: AtomizerVerdict | None    # None when overridden
    decompose: bool                              # resolved decision
    plan: Plan | None                            # None when atomic

    # — new (fan-out reducer) —
    sub_reports: Annotated[list[SubReport], operator.add]  # one per Executor

    # — existing (post-Aggregator) —
    chunks, report, verification, convo, attempts, markdown
```

Two important details:

1. **`sub_reports` is a reducer field** (`Annotated[..., operator.add]`). LangGraph merges concurrent Executor writes into a single list — this is the only safe way to fan-in concurrent state from `Send`-spawned nodes.
2. **The post-Aggregator state mirrors v0.2.1 exactly.** Once the Aggregator writes `chunks` and `report`, the `verify → repair → assemble` chain runs unchanged. This is the swap-invisibility property carried forward: any caller (or test) of `research()` sees a v0.2.1-shaped `ResearchResult`.

## Atomizer (LLM, judge model)

One LLM call on the **judge model** (`qwen2.5:7b-instruct` per current config — small + fast). JSON-mode, Pydantic-validated.

Schema:

```python
class AtomizerVerdict(BaseModel):
    decompose: bool
    rationale: str  # one sentence; logged + included in execution log for debug
```

System prompt (sketch):

```
You decide whether a research question should be broken down. Return JSON
{decompose: bool, rationale: string}.

Decompose if:
- the question covers multiple distinct topics, entities, or time periods,
- the question is a research spec / multi-paragraph brief rather than a
  single question,
- answering it well requires retrieving from multiple unrelated parts of a
  knowledge vault.

Otherwise return decompose=false. Bias toward false when in doubt — a wrong
"true" wastes downstream LLM calls; a wrong "false" produces a still-useful
single-pass answer.
```

User prompt: `f"QUESTION: {query}"`.

**Override surface:** the orchestrator's `/research` endpoint accepts `decompose: Literal["auto"] | bool = "auto"`. `True`/`False` skips the LLM call; `"auto"` calls the Atomizer. This is the escape hatch when the Atomizer guesses wrong on a specific query.

**Rationale for LLM over heuristic** (per sign-off): semantic decisions ("is this a multi-topic spec?") outperform string heuristics on the exact case the user already uses — multi-paragraph spec notes loaded via the plugin's "use note as input" feature. The 7B judge model is fast enough that 1 extra call per research run is acceptable budget.

## Planner prompt + JSON schema

Single LLM call via the inference adapter (synthesis model, JSON mode, Pydantic-validated `Plan`).

System prompt (sketch):

```
You are a research planner. Given a user query, decompose it into the smallest
set of focused sub-queries that, answered independently against a personal
knowledge vault, would let a downstream agent compose a complete answer.

HARD RULES:
- Output JSON conforming to the supplied schema. No prose.
- Each sub-query is a self-contained question — do not reference siblings or
  use pronouns that depend on other sub-queries.
- Each sub-query has a one-sentence rationale.
- Maximum {PLANNER_FANOUT_CAP} sub-queries. Prefer fewer.
- If the query is already atomic, return exactly one sub-query equal to the input.
- Each sub-query target is "vault" (web routing arrives in a later iteration).
```

User prompt: `f"USER QUERY: {query}"`.

`PLANNER_FANOUT_CAP` default = 5. Eval data may push this up/down.

## Executor body (per-Executor verify + repair, per sign-off)

One Executor invocation = the full v0.2.1 spine minus assemble:

```python
async def _execute(state: ExecutorState, config: RunnableConfig) -> dict:
    deps = _deps(config)
    sub_query = state["sub_query"]
    chunks = await deps.retriever.retrieve(sub_query.text, k=state["k"])
    initial = await synthesise(sub_query.text, chunks, deps.client)
    outcome = await repair_loop(
        initial,
        sub_query.text,
        chunks,
        deps.store,
        deps.client,
        max_repair_attempts=state["max_repair_attempts"],
        skip_alignment=state["skip_alignment"],
    )
    return {
        "sub_reports": [
            SubReport(
                sub_query=sub_query,
                chunks=chunks,
                report=outcome.report,
                verification=outcome.verification,
                attempts=outcome.attempts,
            )
        ]
    }
```

`citation.repair_loop` is reused as-is — this is exactly the same function the v0.2.1 orchestrator imported but no longer calls. Pulling it back into Executor scope is correct: each Executor produces a self-contained Report that *should* be verifiable in isolation against its own retrieved chunks.

**Why per-Executor verify+repair (per sign-off, original draft rejected):** the per-Executor repair budget is local — a runaway Executor on one sub-topic doesn't burn the whole budget for the rest. Failed sub-Reports surface to the Aggregator with their `failures` list populated; the Aggregator decides whether to drop them, include with warnings, or pass them upward to the post-merge verify cycle for one more repair attempt against the merged context.

**Post-merge verify+repair still runs.** Its job is to catch cross-Report inconsistencies (e.g. two sub-Reports cite the same `^id` but make contradictory claims). The default `max_repair_attempts=2` budget applies at *each* level — Executor and post-merge — so worst-case repair count is `2*(N+1)`.

## Aggregator (deterministic merge in v0.2.2)

Input: `list[SubReport]`.
Output: a single `Report` and the union of all retrieved chunks (deduped by `block_id`).

Deterministic strategy:

1. `title`: take the original `query` (truncated). The Planner doesn't get to pick the title — that's a synthesis decision.
2. `summary`: concatenate per-SubReport summaries with one blank line between, prefixed with the sub-query's `rationale` as a sub-heading.
3. `sections`: one section per SubReport, heading = `SubReport.sub_query.text` (or its rationale if shorter), claims = union of that SubReport's section claims (preserving order).
4. `chunks`: union over all `SubReport.chunks`, deduped by `block_id`, ordered by max score seen. This becomes the `chunks` field of the parent state; downstream `assemble()` uses it to resolve `^id` → file path.

Trade-off: this produces a structured-but-not-narrative report — the user gets all the sub-research as labelled sections rather than a single flowing essay. That's arguably *better* for a research report (closer to the spec's "decomposed-then-recomposed" intent), but it's a UX call. If the structure-only merge reads poorly in daily use, v0.2.3 promotes the Aggregator to an LLM call that re-writes the merged Report into single-narrative form.

## Diagnostic shim (addresses user's "irrelevant citations" observation)

Independent of the ROMA work, add retrieval visibility so we can debug noise issues with evidence instead of guesses. Cheap, ships alongside v0.2.2:

1. **Per-Executor retrieval log in the response**: extend `ResearchResponse` with `executions: list[{sub_query, chunks: list[{block_id, relpath, score}]}]`. Plugin writes this into the report frontmatter (or a collapsed `## Debug` section gated by a setting).
2. **Optional**: surface the score distribution per sub-query (`min`, `median`, `max`) so the user can spot Executors that retrieved no high-similarity chunks.
3. Orchestrator log line per Executor: `INFO orchestrator.flows.roma: executor sub_query=... k=20 top_score=0.81 mean_score=0.62`.

This costs ~10 lines of code total but lets the user (and future-us) tell at a glance whether the noise is *retrieval* (low-score chunks dominating context) or *synthesis* (high-score chunks but the model still wandered).

## Verification failures: not addressed in v0.2.2 (deliberate)

The user reported verification failures as an equal pain. Why this plan doesn't fix them:

- Verification failures fall into four kinds (`citation.verify.FailureKind`): `BROKEN_LINK`, `QUOTE_NOT_IN_SOURCE`, `UNSUPPORTED` (judge says quote doesn't support claim), `MISSING_CITATION` (AST found a claim with no link).
- The `UNSUPPORTED` kind is judge-strictness; the `BROKEN_LINK`/`QUOTE_NOT_IN_SOURCE` kinds are synthesis-honesty.
- ROMA changes neither. Tighter sub-queries may reduce *opportunities* for the model to fabricate (less context = fewer tempting unrelated chunks), but the failure modes themselves want separate treatment.

The diagnostic shim above tags every failed claim with the Executor it came from, so v0.2.3 can decide whether to (a) tighten the judge prompt, (b) raise the repair budget, (c) skip alignment for sub-Reports and only run it on the merged Report, or (d) make the synthesis prompt more restrictive.

## Eval strategy

Two new test categories, both runnable without live Ollama:

1. **Unit-level**: extend `apps/orchestrator/tests/test_research_flow.py` (the swap-invisibility tests) with two new tests:
   - `test_atomic_query_skips_planner` — short single-question query → only the synthesis stub is called, one Executor runs, identical-shape `ResearchResult` to v0.2.1.
   - `test_decomposed_query_fans_out` — multi-question query → Planner stub returns 3 sub-queries → 3 Executor stubs called concurrently → Aggregator merges → verify+assemble runs once. `client.calls` == 1 (planner) + 3 (executors) = 4.
2. **Smoke-level** (live Ollama, manual): run the user's actual spec note through both flows and compare retrieval logs side-by-side. Success = noticeably fewer low-score chunks in the merged context.

The roadmap's deliverable-2 success criterion (`"agentic research is at least as accurate as v0.1 single-turn on simple queries, and measurably better on multi-source queries"`) needs an eval dataset that distinguishes simple from multi-source queries. We don't have one yet — building it is a small side-quest. Proposal: hand-write 5 simple + 5 multi-source queries against the Alphero vault, expected-block-ids style, add to `eval/harness/datasets/`.

## Implementation order

Each step is independently committable and individually testable.

1. **State + schemas** (Pydantic `SubQuery`, `Plan`, `SubReport`; extend `ResearchState`). No behaviour change, no graph wiring yet. Unit tests for schemas.
2. **Atomizer heuristic** (`atomize` node, config knobs). Test the heuristic with table-driven cases.
3. **Planner node + prompt** (`plan` node, JSON-mode Pydantic call). Test against a stubbed `InferenceClient` returning a fixed `Plan`.
4. **Executor node** (`execute`). Test in isolation with the existing fake client/retriever.
5. **Aggregator node** (`aggregate`). Pure function: test with hand-crafted SubReport lists.
6. **Graph wiring** with `Send` fan-out, conditional routing. Update `graph.py`. Existing v0.2.1 swap-invisibility tests must still pass unchanged.
7. **New flow tests** (atomic + decomposed paths).
8. **Diagnostic shim** in route + Plugin display.
9. **ADR-0021** written and committed.
10. **Component-map row** updated (ROMA Atomizer/Planner/Executor/Aggregator: `deferred → v0.2` → `in (v0.2.2)`).
11. **Plugin**: surface the per-Executor retrieval log in the report frontmatter and optional debug section.

## Risks + open questions (post-sign-off)

| # | Risk / question | Status |
|---|---|---|
| R1 | Local 14B model is poor at multi-step planning; Planner returns junk sub-queries. | Mitigation: JSON-schema-constrained + repaired on invalid output. Content quality surfaced via diagnostic shim. |
| R2 | **Latency is the headline risk.** Worst-case decomposed path = ~20 LLM calls (1 atomizer + 1 planner + 5 × ≤4 [synth + 2 repairs + verify] + post-merge cycle). Even with parallel fan-out, slowest-Executor wall-clock dominates. | Measure end-to-end in eval. Roadmap budget = 30s; if breached, drop `PLANNER_FANOUT_CAP` default to 3 OR cut per-Executor `max_repair_attempts` to 1. Per-Executor + post-merge verify both running is a sign-off choice — we accept the latency cost. |
| R3 | Deterministic Aggregator produces a stitched-together report that reads as a list of mini-reports, not an essay. | Acceptable for v0.2.2 (decomposition transparency is arguably a feature). LLM Aggregator is a v0.2.3 candidate. |
| R4 | LLM Atomizer false-positives (says "decompose" on a simple query) cost N× more than the heuristic would. | Mitigation: `decompose: false` override on `/research`. Bias prompt toward atomic. Diagnostic shim shows atomizer rationale per run. |
| R5 | Atomizer + Planner are *sequential* before fan-out can start. Two LLM round-trips on every decomposed query before any retrieval happens. | Accepted. The judge model is small/fast (~2s typical), so this is ~5s overhead pre-fan-out. |
| Q1 | Should the plugin's debug section be on by default? | Off by default with a settings toggle. Off keeps the report clean; on is the eval/diagnose mode. |
| Q2 | Aggregator title from original query vs first sub-query rationale? | Original query (preserves user intent). |

## Out of scope, captured for v0.2.3+

- LLM-backed Aggregator (re-writes merged Report into single narrative).
- Recursive ROMA (each sub-query goes through its own Atomizer → Planner → ...).
- Verifier-strictness investigation (judge prompt, repair budget) — informed by diagnostic shim data.
- Eval dataset hand-built for simple vs multi-source queries.
