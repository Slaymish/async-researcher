## Goal

Today every research run starts cold — the orchestrator has no memory of the previous turn, last week's research, or facts it has already verified. The roadmap calls this out as the v0.2 deliverable 3 fix: integrate Mem0 as a persistent, local-first memory layer, queried before the Planner runs so its decomposition is informed by what we already know.

The headline value:
- **Tighter sub-queries** — Planner sees "I have prior facts about X" → focuses sub-queries on the gaps, not on already-known ground.
- **Verified facts compound** — every successful (passing verification) claim becomes a fact in memory, so future runs cite from a growing curated set instead of starting from chunks every time.
- **Unblocks deliverable 5** (Crawl4AI web fetch) — web-sourced facts need somewhere to land with temporal metadata, and that "somewhere" is Mem0.

## Non-goals (explicit deferrals)

1. **No memory write from arbitrary nodes.** v0.2.3 has exactly one fact-add site (post-Aggregator, after a successful verify cycle). v0.2.4+ may add Atomizer-derived signals (user preferences inferred from override patterns) or per-Executor signals.
2. **No memory editing / forgetting UX in v0.2.3.** Mem0 is ADD-only (per spec §"Single-Pass ADD"); the user can wipe the store via a CLI command, but there's no per-fact delete UI. Forgetting is a v0.3+ concern.
3. **No Mem0 cloud / managed mode.** Per ADR-0005 (local-first), Mem0 runs as an embedded library against local stores. Self-hosted Docker is documented as the next escalation tier; cloud mode is rejected.
4. **No Mem0 injection into the Atomizer or Executors** (per sign-off). Only the Planner sees memory pre-context in v0.2.3.
5. **No fact-graph / relationships in v0.2.3.** The spec's "entity matching" tri-signal lands later — Mem0's default config gives us semantic + BM25; entity-graph is a v0.2.4 candidate (requires an embeddings + graph store combo that's heavier to configure).

## Library choice (locked: `mem0ai` SDK)

Per sign-off #2. Costs and how we mitigate them:

| Concern | Mitigation |
|---|---|
| Brings `qdrant-client`, `openai`, `posthog`, `sqlalchemy`, `numpy` as transitives. | Accept; this is the cost of not rolling our own. `numpy`/`sqlalchemy` already present. |
| `openai` SDK pulled in — must NEVER hit api.openai.com. | mem0ai's LLM config points at our local Ollama OpenAI-compatible endpoint. `OPENAI_BASE_URL` env override locks the call destination at the SDK level even if config drifts. |
| `posthog` telemetry — anti-pattern for a local-first project. | mem0ai exposes `MEM0_TELEMETRY=False` (and recent versions have `telemetry: false` in config). Set both, belt-and-suspenders. |
| `qdrant-client` defaults to embedded mode (file-backed) — no Docker required. | Use embedded. Storage path lives under `~/Library/Application Support/ai_os/mem0/` next to `index.duckdb`. |

## ADR-0005 / ADR-0009 trade-off (must be acknowledged in ADR-0022)

ADR-0009 says "all LLM calls go through `packages/inference/client.py`". Mem0ai makes its own LLM calls for fact extraction — they go through mem0ai's internal OpenAI client, **not** through our adapter. This is a real deviation from the chokepoint principle.

**The choice:** accept the deviation, with two compensating constraints:

1. **The endpoint mem0ai talks to is our local Ollama.** ADR-0005 (no vault content to cloud LLMs) is preserved — same endpoint, same model, same machine. The chokepoint argument was about *swap* cost (Phase 7 GPU box), and that still holds: when we flip the Ollama URL to a llama-swap URL, we flip mem0ai's `OPENAI_BASE_URL` env at the same time. One config knob, two consumers.
2. **All `packages/memory` callers from our code use our `Memory` wrapper, not mem0ai directly.** The wrapper is the chokepoint for *our* call sites; mem0ai's internal calls are a known leak we document but don't try to plug.

Alternative considered: write a custom LiteLLM-compatible shim that wraps our `InferenceClient` and configure mem0ai to use it. Rejected: high effort, the shim becomes a long-term maintenance burden, and the surfaces that would benefit (audit logs, retry policy) are not v0.2.3 concerns.

## Package surface (`packages/memory`)

Public API (what callers in `apps/orchestrator` see):

```python
from memory import Memory, Fact, MemoryConfig

cfg = MemoryConfig(
    data_dir=Path("~/Library/Application Support/ai_os/mem0").expanduser(),
    llm_base_url="http://localhost:11434/v1",     # Ollama, OpenAI-compat
    llm_model="qwen2.5:14b-instruct",
    embedder_model="nomic-embed-text",
    embedding_dim=768,
    user_id="hamish",                              # Agent Mode identity
    telemetry=False,
)
mem = Memory(cfg)

# Read: tri-signal retrieval (mem0ai handles fusion internally)
facts: list[Fact] = await mem.recall(query, k=8)

# Write: add a fact derived from agent activity
await mem.add(
    text="The deterministic citation pipeline runs in three stages.",
    metadata={"source": "research_run", "query": "...", "block_ids": ["ai-..."]},
)

# Lifecycle
await mem.aclose()
```

`Fact` is our own dataclass — it carries `text`, `score`, `created_at`, and `metadata`. It does NOT leak mem0ai types into our consumers, so a future swap (custom impl or replacement) is a one-package change.

## Storage + filesystem layout

```
~/Library/Application Support/ai_os/
├── index.duckdb          (existing: vault chunks + embeddings)
└── mem0/                 (new in v0.2.3)
    ├── qdrant/           (embedded vector store)
    └── history.db        (SQLite, mem0ai's history table)
```

All under the same `[storage].data_dir` so backup/migration treats them as one unit. New config block in `config.toml`:

```toml
[memory]
enabled = true
user_id = "hamish"
recall_k = 8                # facts injected into Planner pre-context
```

## Injection point (per sign-off #3: Planner only)

The Planner node currently builds its prompt from `system + f"USER QUESTION: {query}"`. v0.2.3 prepends a "MEMORY" block when `recall(query, k=recall_k)` returns ≥1 fact:

```
MEMORY (facts you already know — cite them by id if they support a sub-query):
[m-1] (2026-04-12) The citation engine has three stages: synth, assemble, verify.
[m-2] (2026-05-01) The user prefers structural-merge reports over LLM-rewritten ones.
...

USER QUESTION: ...
```

If `recall()` returns 0 facts, the block is omitted entirely — first-run behaviour is observationally identical to v0.2.2.

The Atomizer and Executors **do not** see this block. Rationale:
- Atomizer's job is "decompose vs atomic"; memory could bias it but the value/cost ratio is bad on a small judge model.
- Executors need the *chunks* the retriever finds — memory facts and vault chunks are different evidence layers, and mixing them in the Executor's context would distort the synthesizer's "cite from chunks" prompt contract (ADR-0013 hard rules 1+2).

## Fact-add lifecycle (when do facts land in memory?)

**v0.2.3 has exactly one writer**, by design — easier to reason about than scattered writes.

After a research run completes successfully (`verification.failures` is empty OR repair budget exhausted with `pass_rate >= threshold`), the orchestrator calls `mem.add(...)` **once per merged Report**, with the merged summary as the fact text and metadata recording:

- `source: "research_run"`
- `query: <original user query>`
- `decompose: <true|false>`
- `block_ids: [list of ^ids cited]`
- `passed: <bool>`

Why summary-as-fact instead of per-claim adds? Two reasons:
1. **ADD-only churn**: per-claim writes would create dozens of micro-facts per run that overwhelm recall. Summary-level facts are the right grain for "what have I researched?".
2. **Reuses existing trust gate**: the synth+verify pipeline already decided this Report's claims are vault-grounded — we don't re-evaluate at memory-write time.

Failed runs (`pass_rate < threshold`, configurable) are **not** added — we don't want unverified noise polluting the store.

## CLI for Agent Mode init (sign-off Q1)

Spec §"Storage Topologies and Autonomous Initialization" describes the agent autonomously minting its own identity. We implement that as:

```bash
uv run orchestrator-memory init [--user-id NAME]
```

What it does:
1. Creates `~/Library/Application Support/ai_os/mem0/` if missing.
2. Writes the `user_id` to `config.toml` (`[memory].user_id`).
3. Runs a one-off `mem.recall("init")` to materialise the empty stores.
4. Prints the storage path + a "ready" confirmation.

Plus utility subcommands:
- `orchestrator-memory list [--limit N]` — dump recent facts (debug).
- `orchestrator-memory wipe` — delete the `mem0/` directory (with confirmation prompt).

## Eval strategy

Two layers:

1. **Unit-level** (`packages/memory/tests/`):
   - Stub-mem0ai test for `Memory.recall()` shape — given a known store contents, what comes back.
   - Test `Memory.add()` is idempotent on identical inputs (no duplicate facts).
   - Test that mem0ai is configured with the right base_url + telemetry=False (regression guard against ADR-0005 drift).

2. **Flow-level** (`apps/orchestrator/tests/test_roma_flow.py` extensions):
   - `test_planner_prompt_includes_memory_facts_when_recall_nonempty` — stub a Memory that returns 2 facts; assert the prompt the Planner sends contains both facts under a "MEMORY" header.
   - `test_planner_prompt_omits_memory_block_when_recall_empty` — first-run invariance.
   - `test_research_run_adds_summary_fact_on_success` — successful end-to-end run hits `mem.add` exactly once with the merged summary.
   - `test_research_run_skips_memory_add_on_failure` — `pass_rate=0` run does not write.

Live-Ollama smoke test (manual): run the same query twice; second run's Planner prompt should include the first run's summary as a memory fact.

## Risks + open questions

| # | Risk / question | Status |
|---|---|---|
| R1 | mem0ai's ADD pipeline itself calls the LLM (it extracts structured facts from raw text). On local 14B, this adds latency to every research run's tail. | Measure first; if intolerable, disable mem0ai's extraction and pass already-structured `Fact` objects directly. mem0ai has a "raw" mode for this. |
| R2 | Embedded qdrant file-locking under concurrent orchestrator + CLI access. | Single-writer assumption: only the orchestrator process writes. CLI commands acquire an advisory lock or fail fast with a "stop the dev server first" message. |
| R3 | Memory pre-context grows the Planner prompt by ~K * fact_length tokens. With K=8 and 80-token facts, +640 tokens per run on the synth model. | Acceptable. K is config-tunable; eval data tells us the right default. |
| R4 | Posthog telemetry might still leak network calls before the `MEM0_TELEMETRY=False` env is read. | Set the env var via `os.environ.setdefault()` at the top of `packages/memory/__init__.py` (before any mem0ai import). Verify with `respx`/network mock that no posthog domain is contacted. |
| R5 | mem0ai version pinning — we hit a stable 2.0.x; major bumps break the embedding model expectation. | Pin minor (`mem0ai>=2.0,<2.1`). Track upstream changelog when bumping. |
| Q1 | Should `orchestrator-memory init` write to `config.toml` or to a separate `mem0.json`? | Plan picks `config.toml` `[memory]` block — fewer files, consistent with the existing config pattern. |
| Q2 | Default `recall_k`? | Plan: 8. Same logic as the Aggregator's chunk dedup — small enough to fit in Planner context, large enough to surface relevant facts. |
| Q3 | Fact-add threshold for "successful enough to remember"? | **Locked at 0.5** (overrode the recommended 0.8 at sign-off — user prefers faster memory accumulation, accepts noise tradeoff). Tunable via `[memory].add_pass_rate_threshold`. |
| Q4 | Per-run override on plugin? | **Yes** — modal carries a "Save to memory after run" checkbox, default ON. Forwarded as `memory_write: bool` on `/research`. Lets the user run throwaway queries without polluting memory. |

## Out of scope, captured for v0.2.4+

- Entity-graph tri-signal (third leg of mem0's tri-signal — needs an entity extractor + graph store).
- LiteLLM shim to route mem0ai's LLM calls through our `InferenceClient`.
- Per-claim fact extraction (vs summary-level).
- User-facing memory browser UI in the plugin.
- Memory-derived sub-query rewriting (Planner sees facts, but doesn't yet *rewrite* sub-queries to skip already-known ground; that's a follow-up).
- Memory export/import for portability across machines.
